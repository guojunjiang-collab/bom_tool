### 1.3 核心问题

1. **单点风险**：3534 行全在一文件，修改任何页面都可能影响其他页面
2. **代码定位困难**：修改零件管理要在 3500 行文件中跳转到特定行
3. **并发开发冲突**：多人同时修改 pages.js 极易冲突
4. **函数命名污染**：大量 `_` 前缀私有函数与公共函数混在一起
5. **BOM 页面过于庞大**：648 行的 `bom()` 函数本身就需要拆分

## 二、拆分原则

### 2.1 总体原则

1. **页面级拆分**：每个主要功能页面 → 独立 `.js` 文件
2. **模块自洽**：每个文件的函数不调用其他页面文件的函数（单向调用可接受：`pages-parts.js` → `Store`，`pages-parts.js` → `API`，`pages-parts.js` → `UI`）
3. **零破坏性迁移**：拆分后功能完全等价于拆分前，不改变任何业务逻辑
4. **渐进式实施**：分阶段拆分，每拆一个验证一个，不一次性全拆

### 2.2 页面间通信策略

| 场景 | 解决方案 |
|------|---------|
| 零件页面 → BOM 页面（查看引用） | 通过 `Router.navigate('bom')` + `sessionStorage` 传参数 |
| 部件页面 → 零件页面（添加子项） | `_showChildSelector` 内部处理，不跨文件 |
| 同步完成 → 刷新当前页 | `Router.render()` 由 `store.js` 调用 |
| 全局通知 | `UI.toast()` / `UI.alert()` / `UI.confirm()`（已有全局 API） |

**关键决策：不引入 EventEmitter / 消息总线。** 当前系统通过 `Router.navigate()` 和 `sessionStorage` 已可满足页面间通信需求。

## 八、风险与注意事项

### 8.1 高风险点

| 风险 | 描述 | 缓解措施 | 实际情况 |

### 8.2 注意事项

1. **不改变任何函数体内容**：拆分过程只移动代码行，不修改任何业务逻辑
2. **保持 `this` 引用一致**：原来用 `this` 的地方，拆分后用当前模块名（如 `Parts.` 或 `Components.`）
3. **sessionStorage 传参**：跨页面数据传递使用 `sessionStorage.setItem('bom_view_ref', id)`
4. **逐步验证**：每拆一个文件，立即测试对应页面的完整功能，不批量拆分

## 九、与同步功能开发计划的关系

拆分工作与同步功能完善是**并行独立**的两条线：

- **拆分**：解决代码可维护性问题，不改变功能行为
- **同步完善**：修复字段映射、同步逻辑，是功能修复/增强

建议先完成拆分，再进行同步功能完善。这样拆分后的同步修复会更清晰（修改哪个文件一目了然）。
