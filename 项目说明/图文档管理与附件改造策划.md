# 图文档管理与零部件附件改造 策划方案

## 一、背景

当前零部件的附件是固定的4个字段（源文件/图纸/STP/PDF），硬编码在 Part 和 Assembly 模型中，无法灵活扩展。新方案将附件提升为独立的一等实体——**图文档**，零部件附件改为引用关联方式，支持用户自行决定关联数量与类别。

## 二、数据库设计

### 2.1 `documents` 表（图文档主表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID PK | 前端生成 UUID，前后端保持一致 |
| code | String(64) NOT NULL | 用户自定义图文档编号 |
| name | String(255) NOT NULL | 文档名称 |
| doc_type | String(64) | 文档分类，来源→系统设置中的图文档分类字典 |
| version | String(32) default="V1.0" | 版本号 |
| description | Text | 描述 |
| file_name | String(255) | 主附件文件名称（冗余字段，列表显示用） |
| file_id | UUID FK→document_attachments.id ON DELETE SET NULL | 关联的主附件 UUID |
| created_at | DateTime | |
| updated_at | DateTime | |

**唯一约束**：`(code, version)`，与零部件件号+版本一致。

### 2.2 `document_attachments` 表（图文档独立附件表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID PK | 前端生成 UUID |
| document_id | UUID FK→documents.id ON DELETE CASCADE | 所属图文档 |
| file_name | String(255) | 文件名（含扩展名） |
| file_data | LargeBinary | 文件二进制数据 |
| file_size | Integer | 文件大小（字节） |
| created_at | DateTime | |

**说明**：与现有 `attachments` 表完全独立。新增的图文档使用自己的附件表，互不干扰。

### 2.3 `entity_documents` 表（零部件↔图文档关联表）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID PK | 前端生成 UUID |
| entity_type | String(32) NOT NULL | 实体类型：`part` / `component` |
| entity_id | UUID NOT NULL | 零部件 ID |
| document_id | UUID FK→documents.id ON DELETE RESTRICT | 关联的图文档 ID |
| category | String(64) | 用户自定义类别标签，如"图纸"/"说明书"/"工艺文件"等 |
| sort_order | Integer | 排序号 |
| created_at | DateTime | |

### 2.4 图文档分类字典

在现有 `dictionary` 表中增加 `doc_type` 字典类型，用户在系统设置中管理图文档分类（如：图纸/模型/说明书/图片/其他等）。

### 2.5 清理旧字段

Part 和 Assembly 模型删除以下8个字段及其外键：
- `source_file` (String)
- `source_file_id` (UUID, FK→attachments.id)
- `drawing` (String)
- `drawing_id` (UUID, FK→attachments.id)
- `stp` (String)
- `stp_id` (UUID, FK→attachments.id)
- `pdf` (String)
- `pdf_id` (UUID, FK→attachments.id)

对应的 SQL：`ALTER TABLE parts DROP COLUMN ...`，同步清理 `attachments` 表中不再被引用的孤数据。

## 三、后端 API 设计

### 3.1 图文档管理 API（新路由 `routers/documents.py`）

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/documents/` | 列表（分页+搜索，按编号/名称/分类筛选） |
| POST | `/api/documents/` | 创建图文档（前端生成 UUID，body 携带 id） |
| GET | `/api/documents/{id}` | 详情（不含文件数据） |
| PUT | `/api/documents/{id}` | 更新元数据 |
| DELETE | `/api/documents/{id}` | 删除（如有零部件引用则阻止） |

### 3.2 图文档附件 API（在 `routers/documents.py` 中扩展）

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/documents/{id}/attachments` | 上传附件（前端生成 UUID，body 携带 id） |
| GET | `/api/documents/{id}/attachments/{att_id}` | 下载附件（返回文件数据） |
| DELETE | `/api/documents/{id}/attachments/{att_id}` | 删除附件 |
| GET | `/api/documents/{id}/attachments/` | 列出附件列表（不含文件数据） |

上传/替换主附件时，同时更新 documents 表的 `file_name` 和 `file_id`。

### 3.3 实体-文档关联 API（在 `routers/parts.py` / `routers/assemblies.py` 中扩展）

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/parts/{id}/documents` | 获取零件的关联图文档列表 |
| POST | `/api/parts/{id}/documents` | 关联图文档（body: document_id, category, sort_order） |
| PUT | `/api/parts/{id}/documents/{edoc_id}` | 更新关联（修改 category/排序） |
| DELETE | `/api/parts/{id}/documents/{edoc_id}` | 移除关联 |

部件同理：`/api/assemblies/{id}/documents`

## 四、前端设计

### 4.1 导航栏增加"图文档管理"

在 `index.html` 侧边栏中 `BOM管理` 后增加：

```html
<a data-page="documents"><span class="nav-icon">📄</span><span>图文档管理</span></a>
```

在 `router.js` 中新增 `documents` 路由 → `Documents.render(c)`。

**新文件**：`frontend/js/pages-documents.js`

### 4.2 图文档管理页面

- **列表页**：表格展示（编号、名称、分类、版本、文件名称、更新时间），支持搜索筛选
- **创建**：弹窗表单（编号、名称、分类下拉、版本、描述），UUID 前端生成
- **详情/编辑**：上方元数据表单，下方附件区域
- **附件上传**：点击"上传附件"→ 选择文件 → 前端生成 UUID → 上传到后端 → 更新 `file_name`/`file_id`
- **附件列表**：显示文件名、大小、上传时间，提供[下载][删除]按钮
- **删除图文档**：有关联引用时提示"已被 XX 个零部件引用，无法删除"

### 4.3 零部件附件改造

**编辑弹窗改造（pages-parts.js / pages-components.js）：**
- 移除原有的4个固定附件行
- 新增"关联图文档"区域，动态列表
- 每行：类别标签（自由输入文本）+ 图文档编号·名称 + [选择] [移除]
- 点击[选择] → 弹出图文档选择器弹窗（搜索框+列表），选中后填写类别标签
- 底部[添加关联]按钮，可添加任意多个

**详情页改造：**
- 附件区域改为动态列表，按类别分组显示

### 4.4 系统设置扩展

`pages-settings.js` 中的字典管理增加 `doc_type` 分类，用户可增删改图文档分类选项。

### 4.5 UUID 一致策略

- 前端 `_uuid()` 生成 UUID v4
- 创建图文档/上传附件时，API body 中携带 `id` 字段
- 后端使用该 `id` 创建记录（与当前 Attachment 的 `id` 处理逻辑一致）

### 4.6 同步机制

图文档使用实时同步（每次操作直接调 API），不做本地离线缓存。`store.js` 中不新增 `documents` 集合，但 `entity_documents` 关联数据随 Part/Component 同步纳入本地存储。

## 五、数据流示意

```
场景：创建一个钣金件零件，关联一份图纸图文档

1. 进入"图文档管理"
   → 上传图纸文件（前端生成 UUID）
   → 后端记录到 document_attachments 表，更新 documents.file_id/file_name

2. 进入"零件管理"，创建/编辑零件
   → 在"关联图文档"区域点击[选择]
   → 弹出图文档选择器，搜索选取刚创建的图文档
   → 输入类别标签：如"图纸"

3. 保存零件
   → entity_documents 表写入：
     entity_type='part', entity_id=xxx, document_id=yyy, category='图纸'
```

## 六、实施步骤

### 阶段一：基础框架（1-2天）

1. 创建 `documents` + `document_attachments` + `entity_documents` 表模型
2. 删除 Part/Assembly 旧附件字段
3. 实现图文档 CRUD API + 附件上传/下载 API
4. 实现图文档前端页面（列表+创建+附件管理）

### 阶段二：关联改造（2-3天）

5. 在 Parts/Components API 中扩展关联接口
6. 改造编辑弹窗关联图文档区域（动态列表+选择器）
7. 改造详情页附件显示
8. 系统设置增加图文档分类字典

### 阶段三：清理（1天）

9. 删除旧附件相关的前端代码（4个固定字段的渲染/上传/下载/删除逻辑）
10. 清理数据库中旧的 `attachments` 表未被引用的数据
11. 端到端测试
