# 零部件附件功能详细设计与开发计划

> 编写日期：2026-04-21
> 版本：v1.0

---

## 一、当前附件功能现状

### 1.1 数据库层：两套方案并存

系统中存在**两套独立的附件存储机制**：

#### 方案 A：实体内联字段（当前前端实际使用的方案）

零部件表（`parts` / `assemblies`）中各有 4 个内联附件字段：

| 字段 | 说明 | 存储方式 |
|------|------|---------|
| `source_file` | 源文件名 | 文本 |
| `source_file_data` | 源文件内容 | Base64（极大） |
| `drawing` | 图纸文件名 | 文本 |
| `drawing_data` | 图纸内容 | Base64（极大） |
| `stp` | STP 文件名 | 文本 |
| `stp_data` | STP 文件内容 | Base64（极大） |
| `pdf` | PDF 文件名 | 文本 |
| `pdf_data` | PDF 文件内容 | Base64（极大） |

#### 方案 B：独立附件表（后端已有，前端未使用）

`attachments` 表（独立表结构）：

| 字段 | 说明 |
|------|------|
| `id` | UUID 主键 |
| `entity_type` | 实体类型：`part` 或 `component` |
| `entity_id` | 关联实体 ID |
| `file_type` | 文件类型：`source_file` / `drawing` / `stp` / `pdf` |
| `file_name` | 文件名 |
| `file_data` | Base64 文件内容 |

### 1.2 后端 API

| 接口 | 方案 | 用途 |
|------|------|------|
| `POST /attachments/` | B | 上传/更新附件（entity_type + entity_id + file_type + Base64） |
| `DELETE /attachments/by-entity` | B | 删除附件（按 entity + file_type） |
| `DELETE /attachments/{id}` | B | 删除附件（按 attachment ID） |
| `GET /attachments/` | B | 查询附件列表（支持过滤） |
| `GET /parts/{id}` | A | 获取零件（含 Base64，include_attachments=true） |
| `PUT /parts/{id}` | A | 更新零件（含 Base64） |
| `GET /assemblies/{id}` | A | 获取部件（含 Base64） |
| `PUT /assemblies/{id}` | A | 更新部件（含 Base64） |

### 1.3 前端实现

**仅使用方案 A（内联字段）**，上传流程：

```
文件选择 → Base64 转换 → 表单提交时一起发送 → 内嵌在零件/部件数据中
```

关键代码逻辑（`pages.js`）：
- 零件/部件表单各有 4 个 `<input type="file">`（源文件/图纸/STP/PDF）
- `UI._fileToBase64()` 将文件转为 Base64
- 保存时判断是否选择了新文件：
  - 有新文件 → 使用新 Base64
  - 无新文件 → 保留本地存储的旧 Base64（`part[k + '_data']`）
- 删除附件 → 调用 `API.deleteAttachment()`（方案 B 的 API）→ 同时清除本地字段

### 1.4 同步实现

| 操作 | 状态 | 说明 |
|------|------|------|
| 上传附件 | ❌ **未同步** | 附件通过内联表单发送，不经过 store.js 同步队列 |
| 删除附件 | ✅ 部分同步 | 先调 API 删除服务器记录，再清除本地字段 |
| 拉取附件 | ⚠️ 仅手动 | 打开编辑页面时从服务器加载（`include_attachments=true`） |
| 同步队列 | ❌ 附件不入队 | `store.js` 不处理 Base64 字段的同步 |

## 二、问题分析

### 2.1 核心问题：两套方案混用，职责不清

**方案 A（内联字段）的问题：**

1. **无法同步**：Base64 数据太大（约 5MB 限制），无法通过同步队列传输
2. **刷新丢失**：上传附件后刷新页面，附件从服务器加载（`include_attachments=true`），但修改零件后再同步，内联 Base64 不被 store.js 处理
3. **存储臃肿**：localStorage 约 5MB 限制，1 个 PDF 文件就能占满
4. **带宽浪费**：每次更新实体都要传所有附件 Base64

**方案 B（独立附件表）的优势：**

1. **职责分离**：附件与实体独立，单独上传/下载
2. **可同步**：附件元数据（file_name/file_type）可纳入同步，文件内容按需加载
3. **原子性**：更新实体不涉及附件，附件更新不影响实体
4. **独立管理**：附件可单独删除、替换

**结论：前端应统一迁移到方案 B**，方案 A 的内联 Base64 字段应废弃（数据库字段保留，数据迁移后可清空）。

### 2.2 具体症状

| 症状 | 根因 |
|------|------|
| 上传附件后，修改零件其他字段，附件消失 | 内联 Base64 不在同步队列中，被服务器覆盖 |
| 刷新页面后附件存在，但再编辑其他字段后消失 | 同上，服务器更新时丢失内联 Base64 |
| 零件列表加载慢 | 所有零件含 Base64 附件，拉取列表时全部下载 |
| 编辑页面附件下载失败 | Base64 存储/传输过程中的编码问题 |

### 2.3 根本原因

`_fieldMap` 中包含了 `source_file / drawing / stp / pdf` 字段名映射，但同步队列**无法传输 Base64**，导致：

```
上传附件 → Base64 存入 part.sourceFile_data → 保存实体
    ↓
同步队列 UPDATE part → API.updatePart() → 服务器保存了 Base64 ✅（首次上传OK）
    ↓
修改其他字段 → 同步队列 UPDATE part → 
    → 表单数据中的 sourceFile_data = ''（因为没选文件）
    → API.updatePart() → 服务器 source_file_data 被覆盖为空 ❌
```

## 三、详细设计方案

### 3.1 架构决策

**统一采用方案 B（独立附件表）**，内联 Base64 字段逐步废弃：

| 阶段 | 内联字段处理 | 附件同步方式 |
|------|------------|------------|
| 当前 | 全部保留 | 不处理 |
| 过渡期 | 前端停止写入 | 前端改用独立附件 API |
| 最终 | 数据库字段保留，数据迁移后清空 | 完全由附件 API 管理 |

### 3.2 附件元数据同步设计

附件元数据（不含 Base64）纳入同步流程：

```javascript
// 方案 B 附件元数据结构
{
  entity_type: 'part',       // 'part' | 'component'
  entity_id: 'xxx',          // 关联实体 ID
  file_type: 'drawing',      // 'source_file' | 'drawing' | 'stp' | 'pdf'
  file_name: '图纸_v1.pdf',  // 文件名
  file_id: 'uuid',           // 附件表主键（用于精确删除）
  has_local: true,            // 本地是否有 Base64 缓存
}
```

#### 3.2.1 字段映射

```javascript
attachments: {
  entityType: 'entity_type',
  entityId: 'entity_id',
  fileType: 'file_type',
  fileName: 'file_name',
  fileId: 'id',
  createdAt: 'created_at',
  updatedAt: 'updated_at',
},
```

#### 3.2.2 同步流程

**附件元数据同步（不入同步队列，单独处理）：**

```
【上传附件】
1. 选择文件 → Base64 编码
2. 调用 API.uploadAttachment(entity_type, entity_id, file_type, file_name, file_data)
3. 服务器保存到 attachments 表，返回 attachment_id
4. 本地记录 attachment_id（用于后续精确删除/替换）
5. 可选：将 attachment_id 写入实体的 _attachments 字段，触发实体同步更新元数据

【删除附件】
1. 调用 API.deleteAttachment(entity_type, entity_id, file_type)
2. 服务器删除 attachments 表记录
3. 本地清除 attachment_id 记录

【下载附件】
1. 调用 API.getAttachment(attachment_id) 获取 Base64
2. UI._downloadBase64(base64, fileName)
```

### 3.3 实体同步中附件的处理

**修改后的同步流程：**

```javascript
// store.js _execSync 中，update 流程修改：
if (entity === 'parts') {
  // 不再发送 source_file_data/drawing_data/stp_data/pdf_data
  const { source_file_data, drawing_data, stp_data, pdf_data, ...rest } = apiData;
  await API.updatePart(resolvedId, rest);  // 不含 Base64
  // 附件通过独立 API 管理，不走同步队列
} else if (entity === 'components') {
  const { source_file_data, drawing_data, stp_data, pdf_data, ...rest } = apiData;
  await API.updateAssembly(resolvedId, rest);
}
```

### 3.4 按需加载策略

附件不随实体列表一起加载，而是按需获取：

```javascript
// 实体列表页：只显示附件文件名，不含 Base64
// 实体详情页/编辑页：按需加载附件
async function loadAttachments(entity_type, entity_id) {
  const attachments = await API.listAttachments(entity_type, entity_id);
  // attachments = [{ id, file_type, file_name, ... }]
  return attachments;
}

// 点击"下载"时获取 Base64
async function downloadAttachment(attachment_id) {
  const att = await API.getAttachment(attachment_id);
  UI._downloadBase64(att.file_data, att.file_name);
}
```

### 3.5 升版时附件的处理

| 方案 | 描述 | 决策 |
|------|------|------|
| 方案 1 | 升版时**不复制**附件，新版本从空白开始 | **推荐**：不同版本可能有不同的源文件/图纸 |
| 方案 2 | 升版时**复制**所有附件到新版本 | 不推荐：新版本的附件可能不同 |

**决策：升版时附件不自动复制**，新版本从空白开始，由用户按需上传。

#### 4.2.1 api.js — 补充附件 API

```javascript
// 现有（需补充）：
async deleteAttachment(entity_type, entity_id, file_type) { ... }  // ✅ 已有

// 新增：
async listAttachments(entity_type, entity_id) {
  return this._fetch('GET', '/attachments/?entity_type=' + entity_type + '&entity_id=' + entity_id);
},

async uploadAttachment(entity_type, entity_id, file_type, file_name, file_data) {
  return this._fetch('POST', '/attachments/', {
    entity_type, entity_id, file_type, file_name, file_data
  });
},

async getAttachment(attachment_id) {
  return this._fetch('GET', '/attachments/' + attachment_id);

  // 获取附件并触发下载
  const atts = await this.listAttachments(entity_type, entity_id);
  const att = atts.find(a => a.file_type === file_type);
  if (att) {
    const full = await this.getAttachment(att.id);
    UI._downloadBase64(full.file_data, full.file_name);
  }
},
```

### 4.3 Phase 2：前端附件上传/删除迁移（3 小时）

**目标**：修改 `pages.js`，将附件操作从内联方式迁移到独立 API。

#### 4.3.1 零件表单附件处理

**现状逻辑**（`pages.js` `_savePart`）：
```javascript
// 提交时，将所有 Base64 一起发送
fileKeys.forEach(function(k) {
  var r = results[k];  // 是否有新文件
  if (r) {
    data[k] = r.name;
    data[k + '_data'] = r.data;  // 新 Base64
  } else {
    data[k] = part ? (part[k] || '') : '';
    data[k + '_data'] = part ? (part[k + '_data'] || '') : '';  // 旧 Base64
  }
});
Store.update('parts', partId, data);
```

**修改后的逻辑**：
```javascript
// 提交时，不含 Base64
fileKeys.forEach(function(k) {
  data[k] = part ? (part[k] || '') : '';
  // 不再设置 data[k + '_data']
});
Store.update('parts', partId, data);

// 保存后，单独上传新选择的附件
Promise.all(
  files.map(function(f, i) {
    var fileInput = document.getElementById(f);
    if (fileInput && fileInput.files[0]) {
      var file = fileInput.files[0];
      return UI._fileToBase64Async(file).then(function(base64) {
        return API.uploadAttachment('part', partId, fileKeys[i], file.name, base64);
      });
    }
    return Promise.resolve();
  })
).then(function() {
  UI.toast('保存并上传附件成功', 'success');
  renderParts();
});
```

#### 4.3.2 零件编辑页附件显示

**现状**：从 `part.sourceFile_data` 读取 Base64。

**修改后**：从 `API.listAttachments('part', partId)` 获取附件列表。

```javascript
async function refreshPartAttachments(partId) {
  var attachments = await API.listAttachments('part', partId);
  var attMap = {};
  attachments.forEach(function(att) {
    attMap[att.file_type] = att;  // { source_file: {...}, drawing: {...}, ... }
  });
  
  // 更新预览区
  ['sourceFile','drawing','stp','pdf'].forEach(function(field) {
    var att = attMap[field] || null;
    var preview = document.getElementById('fp-' + field.toLowerCase()).parentElement.querySelector('.file-preview');
    if (att) {
      preview.innerHTML = '<span class="file-name">' + _esc(att.file_name) + '</span>' +
        '<button onclick="downloadAttachment(\'' + att.id + '\')">下载</button>' +
        '<button onclick="deleteAttachment(\'' + att.entity_type + '\',\'' + att.entity_id + '\',\'' + att.file_type + '\')">删除</button>';
    } else {
      preview.innerHTML = '<span class="file-name empty">未上传</span>';
    }
  });
}
```

#### 4.3.3 附件删除逻辑

**现状**：调用 `API.deleteAttachment()` + 清除本地字段。

**修改后**：保持调用 `API.deleteAttachment()`，但清除本地附件列表缓存。

```javascript
async function deleteAttachment(entity_type, entity_id, file_type) {
  await API.deleteAttachment(entity_type, entity_id, file_type);
  // 清除本地缓存的附件信息
  delete window._partAttachmentsCache[entity_id + '_' + file_type];
  // 刷新预览
  if (entity_type === 'part') refreshPartAttachments(entity_id);
  else refreshCompAttachments(entity_id);
}
```

#### 4.3.4 同样的逻辑应用于部件（components）

将零件的附件处理逻辑同样应用于部件的 `pages.js`：
- `_onCompFileChange()` 改为上传到独立 API
- `_saveComp()` 中移除 Base64 字段
- `_compAttachmentView` 改为从 API 加载附件

### 4.4 Phase 3：同步队列适配（1 小时）

**目标**：确保 `store.js` 不传输 Base64，附件独立管理。

if (entity === 'parts') {
  // 过滤掉所有 _data 字段

## 七、风险与注意事项

### 7.1 风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 过渡期内联/独立两套并存 | 数据不一致 | 前端迁移完成后统一，后端保留两套接口 |
| Base64 编码问题 | 附件损坏 | 上传/下载使用统一的 Base64 编解码 |
| 大文件上传超时 | 上传失败 | 前端增加上传进度提示，后端增加超时配置 |
| 并发上传同一 file_type | 覆盖而非追加 | 当前设计为覆盖；Phase 5 可改为多文件 |

### 7.2 注意事项

1. **不需要修改数据库**：attachments 表已存在，无需 DDL
2. **无需修改后端**：后端 API（`attachments.py`）已完整实现
3. **前端修改影响面广**：涉及零件/部件的新建、编辑、查看、删除多个页面
4. **测试重点**：确保"修改实体其他字段不影响已有附件"这一核心场景通过

## 八、与同步功能开发计划的关系

附件功能属于 **独立于通用同步框架的特殊数据类型**，需要单独处理：

| 方面 | 通用同步（Parts/Components） | 附件同步 |
|------|---------------------------|---------|
| 数据规模 | 小（KB级） | 大（MB级） |
| 同步方式 | 队列静默同步 | 独立 API，即时操作 |
| Base64 处理 | 过滤，不传输 | 独立 API 传输 |
| 列表加载 | 随列表加载 | 按需加载（不进列表） |
