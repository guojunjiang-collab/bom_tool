## 🚀 主要功能实现

### 1. 部件子项同步机制修复
**目标**：解决部件子项（BOM items）同步后丢失的问题，确保数据持久化
- **问题根因**：`child_id` 指向字典表（dict_materials/dict_units等）的本地临时ID，但字典表在后台同步队列中被跳过（skip list），导致 `resolveId` 永远返回 `null`
- **修复方案**：在 `_execSync` 中添加验证逻辑，跳过无效子项，并输出详细日志定位问题
- **同步流程优化**：`_enqueue` 操作后自动标记 `_needsRefresh`，同步完成时触发 `Router.render()` 刷新页面

### 2. 后端数据库子项数据类型统一

- **插入逻辑**：在 `add_assembly_part` 路由中添加转换：`if item.child_type == 'assembly': item.child_type = 'component'`
- **查询逻辑**：`get_assembly_parts` 函数支持 `child_type` 为 `component` 或 `part`，返回本地格式字段 `childType`、`componentId`、`partId`
- **重启服务**：执行 `docker restart bom_backend` 使更改生效

### 5. 材料字段显示修复
**目标**：解决零件管理表格和详情页材料字段显示空白或UUID的问题
- **根因分析**：
  1. `_fieldMap['dict_materials']` 缺少 `id` 映射，导致 `Store.getById` 查找失败
  2. `_fieldMap['parts']` 中存在错误映射 `material: 'material_id'`，导致反向映射时覆盖名称字段
  3. 零件编辑后未从服务器重新拉取最新数据
- **修复方案**：
  1. 为 `dict_materials` 添加 `id: 'id'` 映射
  2. 移除 `parts` 中的 `material: 'material_id'` 映射
  3. 直接使用 `part.material`（服务器返回的名称字段）显示
  4. 同步成功后主动调用 `API.getPart` 更新本地存储

## 🐛 关键Bug修复
1. **部件添加子项后 422 错误**：`child_id` 为空字符串
   - **原因**：`_editComp` 保存时子项数据结构转换错误，`componentId` 被放入 `partId` 字段
   - **修复**：区分零件/部件类型分别保留正确的 ID 字段，同步时添加无效子项检查

2. **部件详情中子项不显示**：树形视图和表格视图均为空
   - **原因**：保存时的 filter 条件 `return p.partId` 过滤掉了所有 `childType === 'component'` 的子项
   - **修复**：改为 `return (p.childType === 'component' ? p.componentId : p.partId)`

3. **零件编辑后材料属性不更新**：修改材料后界面仍显示旧值
   - **原因**：同步完成后未从服务器重新拉取最新数据，本地缓存未更新
   - **修复**：`_processQueue` 中 update 成功后主动调用 `API.getPart` 并合并到本地存储

4. **BOM items 引用无法跟随实体 ID 更新**（跨会话遗留问题）
   - **原因**：`_updateLocalId` 无法匹配 BOM items 中的旧引用，因为 `oldId` 已被 `Object.assign` 覆盖为服务器 UUID
   - **待修复方案**：需在 `Store.add` 中保存原始 localId，作为第四参数传入 `_updateLocalId`

5. **字典表 ID 映射缺失**：清除缓存重新同步后材料字段显示异常
   - **原因**：`dict_materials` 缺少 `id` 映射，导致本地存储的字典项无 `id` 字段
   - **修复**：在 `_fieldMap` 中添加 `id: 'id'` 映射
