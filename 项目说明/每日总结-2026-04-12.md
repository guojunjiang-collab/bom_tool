### 2. 修订记录功能（4月12日）
**目标**：记录零部件当前版本内的每次字段修改明细

**设计原则**：
- **修订记录（revisions）**：记录当前版本内所有字段修改的详细变更
- **版本历史（versionHistory）**：已移除，仅保留修订记录

**追踪字段**（零件/部件相同）：
- `code`（件号）、`name`（名称）、`spec`（规格型号）
- `status`（状态）、`remark`（备注）
- `sourceFile`（源文件）、`drawing`（图纸）、`stp`（STP）、`pdf`（PDF）

**数据结构**：
```javascript
revisions: [{
  date: 1712937600000,           // 修改时间戳
  author: "张工程师",             // 修改人
  changes: [
    { field: "名称", oldVal: "M12螺栓", newVal: "M12螺栓-加长" },
    { field: "规格型号", oldVal: "M12×50", newVal: "M12×80" }
  ]
}]
```

**记录时机**：
| 场景 | 记录内容 |
|------|---------|
| 编辑保存 | 对比9个字段，仅记录变化的字段 |
| 新增 | 初始化 `revisions: []` |
| 升版 | 记录 `changes: [{field: '版本', oldVal: 'A', newVal: 'B'}]` |

**UI展示**：
- `_viewPart`/`_viewComp` 底部始终显示「📝 修订记录」区域
- 有记录：按时间倒序，显示修改人、时间、字段变更对比（红色旧值→绿色新值）
- 无记录：灰色提示「暂无修订记录」

**后端支持**：
- `Part.revisions` → JSONB 列，默认 `[]`
- `Assembly.revisions` → JSONB 列，默认 `[]`

### 3. 清除缓存功能（4月12日）
**目标**：提供一键清除本地缓存，保留登录状态

**功能位置**：顶部工具栏，同步数据按钮右侧

**实现逻辑**：
```javascript
// 保留的键
keepKeys = ['bom_api_token', 'bom_current_user']

// 包括：bom_parts, bom_components, bom_users, bom_logs 等
```

**Bug修复**：
- 原代码错误使用 `bom_user`，实际键名为 `bom_current_user`
- 修复后清除缓存不再误删用户信息，无需重新登录

### 4. 同步字段映射修复（4月12日）
**问题**：同步后零件详情未显示规格型号等数据

**根因**：`Store._fieldMap.parts` 缺少 `createdAt`/`updatedAt` 映射

**修复**：
```javascript
// 修改前
parts: { remark: 'remark', lossRate: 'loss_rate', revisions: 'revisions' }

// 修改后  
parts: { 
  remark: 'remark', 
  lossRate: 'loss_rate', 
  revisions: 'revisions',
  createdAt: 'created_at', 
  updatedAt: 'updated_at' 
}
```

### 5. 升版同步Bug修复（4月12日）
**问题**：升版零件后同步数据报错，新零件上传失败

**根因**：后端 `parts.code` 有唯一约束，但前端升版设计是**同编码多版本**（A→B→C）

**修复内容**：

| 文件 | 改动 |
|------|------|
| `models.py` | 添加 `version` 字段，改为 `(code, version)` 联合唯一约束 |
| `schemas.py` | `PartBase`/`PartCreate`/`PartUpdate` 添加 `version` 字段 |
| `crud.py` | `get_part_by_code(db, code, version=None)` 支持按版本查询 |
| `parts.py` | 创建零件时检查 `(code, version)` 联合唯一 |
| 数据库 | 添加 `version` 列，删除旧唯一约束，添加联合唯一约束 |

**SQL变更**：
```sql
ALTER TABLE parts ADD COLUMN version VARCHAR(32) DEFAULT 'A';
ALTER TABLE parts DROP CONSTRAINT parts_code_key;
ALTER TABLE parts ADD CONSTRAINT uix_part_code_version UNIQUE (code, version);
```

## 📊 技术架构更新

### 前端架构
- **同步机制**：手动全量同步 + 后台实时队列双轨并行
- **字段映射**：`Store._fieldMap` 统一管理前后端字段名转换
- **权限控制**：`Auth.canEdit()` / `Auth.isAdmin()` 统一入口
- **状态管理**：`draft`/`frozen`/`released`/`obsolete` 四态工作流

### 后端架构
- **模型关系**：
  ```
  Part (code, version) 联合唯一
  Assembly (code, version) 联合唯一  
  BOMItem → part_id / assembly_id
  ```
- **API规范**：RESTful + snake_case 字段名
- **数据库**：PostgreSQL + JSONB 存储修订记录

## 📈 数据流说明

### 同步数据流程
```
用户点击同步
    ↓
SyncModal.start()
    ↓
┌─────────────────────────┐
│ 第一阶段：拉取（5-30%）   │
│ 遍历3类数据：             │
│ parts / components / users│
│   ↓                      │
│ API.getXxx() 拉取全量     │
│   ↓                      │
│ reverseMap 字段转换       │
│ （snake_case→camelCase）  │
│   ↓                      │
│ Upsert 本地 Store        │
└─────────────────────────┘
    ↓
┌─────────────────────────┐
│ 第二阶段：上传（40-95%） │
│ 找出「本地有服务器无」     │
│   ↓                      │
│ fmap 字段转换             │
│ （camelCase→snake_case）  │
│   ↓                      │
│ 分类上传                  │
│ · 部件：主体→子项        │
│ · 其他：直接上传          │
└─────────────────────────┘
```

### 后台实时队列
```
用户操作 → Store._enqueue() → _syncQueue[] 
    ↓
_processQueue() → _execSync(task)
    ↓
┌────────────────────────────────────────┐
│ add: toApi() → API.create →            │
│     _updateLocalId 替换临时ID →        │
│     部件子项关联                        │
├────────────────────────────────────────┤
│ update: toApi() → API.update →         │
│     部件：先主体后子项                  │
├────────────────────────────────────────┤
│ delete: API.delete(id) →               │
│     部件：先删子项再删主体              │
└────────────────────────────────────────┘
```
