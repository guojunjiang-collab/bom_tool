## 二、当前同步方案分析

### 2.1 整体架构

系统采用**本地优先 + 后台静默同步**架构：

```
用户操作 → 立即写入 localStorage → 后台同步队列 → 服务器
                ↓                              ↓
           UI 立即更新                   异步推送/拉取
```

**核心组件：**

- `Store._syncQueue`：待同步任务队列（FIFO）
- `Store._enqueue()`：本地操作完成后自动入队
- `Store._processQueue()`：串行执行同步任务（非阻塞）
- `Store._getIdMap()`/`_setIdMap()`：本地临时ID ↔ 服务器真实ID 映射
- `Store.resolveId()`：解析服务器真实ID
- `Store.registerId()`：注册 ID 映射关系

### 2.2 当前各模块同步方案

#### 2.2.1 零件（Parts）同步

| 操作         | 同步方式                        | 冲突检测                  |
| ---------- | --------------------------- | --------------------- |
| 新建（add）    | API.createPart → 服务器生成 UUID | 检查 code+version 是否冲突  |
| 修改（update） | API.updatePart              | 比较 updatedAt，服务器更新则拒绝 |
| 删除（delete） | API.deletePart              | 无                     |
| 升版         | 无（升版视为新建）                   | 检查 code+version       |

**现状：**

- 同步完成后，从服务器拉取最新数据更新本地（覆盖本地）
- 字段映射：`_fieldMap['parts']` 只包含 `status/remark/revisions/createdAt/updatedAt`
- **缺失同步的字段**：name、spec、material_id、price、stock、attachments

#### 2.2.2 部件（Components）同步

| 操作         | 同步方式                                          | 说明                          |
| ---------- | --------------------------------------------- | --------------------------- |
| 新建（add）    | API.createAssembly → 添加子项 → 重新拉取 BOM item IDs | 子项通过 `addAssemblyPart` 逐个上传 |
| 修改（update） | API.updateAssembly → **三步同步子项**               | 详见 1.2.2.1                  |
| 删除（delete） | API.deleteAssembly（级联删除子项）                    | 后端 cascade 处理               |
| 升版         | 无（升版视为新建）                                     | 检查 code+version             |

**1.2.2.1 部件子项三步同步算法（当前实现）：**

```
1. 新增子项：遍历本地 parts，serverChildIds 不存在 → addAssemblyPart
2. 删除子项：遍历服务器 parts，本地 childId 不存在 → removeAssemblyPart(sp.id) 优先，
   或 removeAssemblyPartByChild(sp.child_id, sp.child_type) 回退
3. 修改数量：遍历服务器 parts，数量不同 → 先删除再添加
```

**关键机制：**

- `localChildIds`：通过 `resolveId()` 解析子项 ID 后构建的 Set
- `bomItemId`：服务器 BOM item 的 UUID，存储在本地 `parts[].bomItemId` 字段
- 子项 ID 映射：本地子项的 `componentId/partId` 可能为临时 ID，通过 `resolveId` 解析

**缺失同步的字段：** name、spec、price、attachments

#### 2.2.3 用户（Users）同步

| 操作         | 同步方式           | 冲突检测             |
| ---------- | -------------- | ---------------- |
| 新建（add）    | API.createUser | 检查 username 是否冲突 |
| 修改（update） | API.updateUser | **无冲突检测**        |
| 删除（delete） | API.deleteUser | 无                |

**现状：**

- 同步完成后**不拉取**最新数据到本地
- 字段映射：`_fieldMap['users']` 包含 `realName/department/phone/createdAt/updatedAt`
- **缺失同步的字段**：role、status

#### 2.2.4 系统字典（Dictionary）同步

**当前状态：完全未实现**

- `_fieldMap['dict_materials']` 存在但**未在 `_entityMap` 中注册**
- 无 CRUD 同步逻辑
- 字典数据仅在登录后从服务器拉取一次到内存

## 三、附件功能现状

### 3.1 数据库层：两套方案并存

系统中存在**两套独立的附件存储机制**：

#### 方案 A：实体内联字段（当前前端实际使用的方案）

零部件表（`parts` / `assemblies`）中各有 4 个内联附件字段：

| 字段                 | 说明       | 存储方式       |
| ------------------ | -------- | ---------- |
| `source_file`      | 源文件名     | 文本         |
| `source_file_data` | 源文件内容    | Base64（极大） |
| `drawing`          | 图纸文件名    | 文本         |
| `drawing_data`     | 图纸内容     | Base64（极大） |
| `stp`              | STP 文件名  | 文本         |
| `stp_data`         | STP 文件内容 | Base64（极大） |
| `pdf`              | PDF 文件名  | 文本         |
| `pdf_data`         | PDF 文件内容 | Base64（极大） |

#### 方案 B：独立附件表（后端已有，前端未使用）

`attachments` 表（独立表结构）：

| 字段            | 说明                                             |
| ------------- | ---------------------------------------------- |
| `id`          | UUID 主键                                        |
| `entity_type` | 实体类型：`part` 或 `component`                      |
| `entity_id`   | 关联实体 ID                                        |
| `file_type`   | 文件类型：`source_file` / `drawing` / `stp` / `pdf` |
| `file_name`   | 文件名                                            |
| `file_data`   | Base64 文件内容                                    |

### 3.2 后端 API

| 接口                              | 方案  | 用途                                                    |
| ------------------------------- | --- | ----------------------------------------------------- |
| `POST /attachments/`            | B   | 上传/更新附件（entity_type + entity_id + file_type + Base64） |
| `DELETE /attachments/by-entity` | B   | 删除附件（按 entity + file_type）                            |
| `DELETE /attachments/{id}`      | B   | 删除附件（按 attachment ID）                                 |
| `GET /attachments/`             | B   | 查询附件列表（支持过滤）                                          |
| `GET /parts/{id}`               | A   | 获取零件（含 Base64，include_attachments=true）               |
| `PUT /parts/{id}`               | A   | 更新零件（含 Base64）                                        |
| `GET /assemblies/{id}`          | A   | 获取部件（含 Base64）                                        |
| `PUT /assemblies/{id}`          | A   | 更新部件（含 Base64）                                        |

### 3.3 附件同步现状

| 操作   | 状态        | 说明                                        |
| ---- | --------- | ----------------------------------------- |
| 上传附件 | ❌ **未同步** | 附件通过内联表单发送，不经过 store.js 同步队列              |
| 删除附件 | ✅ 部分同步    | 先调 API 删除服务器记录，再清除本地字段                    |
| 拉取附件 | ⚠️ 仅手动    | 打开编辑页面时从服务器加载（`include_attachments=true`） |
| 同步队列 | ❌ 附件不入队   | `store.js` 不处理 Base64 字段的同步               |

## 四、问题汇总

### 4.1 字段同步不完整

| 模块         | 缺失字段                                               | 影响             |
| ---------- | -------------------------------------------------- | -------------- |
| Parts      | name, spec, material_id, price, stock, attachments | 修改这些字段不会同步到服务器 |
| Components | name, spec, price, attachments                     | 修改这些字段不会同步到服务器 |
| Users      | role, status                                       | 修改这些字段不会同步到服务器 |

### 4.2 附件核心问题：两套方案混用，职责不清

**方案 A（内联字段）的问题：**

1. **无法同步**：Base64 数据太大（约 5MB 限制），无法通过同步队列传输
2. **刷新丢失**：上传附件后刷新页面，附件从服务器加载（`include_attachments=true`），但修改零件后再同步，内联 Base64 不被 store.js 处理
3. **存储臃肿**：localStorage 约 5MB 限制，1 个 PDF 文件就能占满
4. **带宽浪费**：每次更新实体都要传所有附件 Base64

**方案 B（独立附件表）的优势：**

1. **职责分离**：附件与实体独立，单独上传/下载
2. **可同步**：附件元数据（file_name/file_type）可纳入同步，文件内容按需加载
3. **原子性**：更新实体不涉及附件，附件更新不影响实体
4. **独立管理**：附件可单独删除、替换

**结论：前端应统一迁移到方案 B**，方案 A 的内联 Base64 字段应废弃（数据库字段保留，数据迁移后可清空）。

### 4.3 其他问题

- 升版功能缺失
- 冲突检测不完善
- 字典同步缺失
- 拉取同步缺失

#### 5.3.3 同步流程

**附件元数据同步（不入同步队列，单独处理）：**

```
【上传附件】
1. 选择文件 → Base64 编码
2. 调用 API.uploadAttachment(entity_type, entity_id, file_type, file_name, file_data)
3. 服务器保存到 attachments 表，返回 attachment_id
4. 本地记录 attachment_id（用于后续精确删除/替换）
5. 可选：将 attachment_id 写入实体的 _attachments 字段，触发实体同步更新元数据

【删除附件】
1. 调用 API.deleteAttachment(entity_type, entity_id, file_type)
2. 服务器删除 attachments 表记录
3. 本地清除 attachment_id 记录

【下载附件】
1. 调用 API.getAttachment(attachment_id) 获取 Base64
2. UI._downloadBase64(base64, fileName)
```

### 5.5 按需加载策略

附件不随实体列表一起加载，而是按需获取：

```javascript
// 实体列表页：只显示附件文件名，不含 Base64
// 实体详情页/编辑页：按需加载附件

  const attachments = await API.listAttachments(entity_type, entity_id);
  // attachments = [{ id, file_type, file_name, ... }]
  return attachments;

// 点击"下载"时获取 Base64

```

### 5.6 冲突检测完善

#### 5.6.1 时间戳冲突

所有 update 操作增加 `updatedAt` 冲突检测：

```
1. 同步前获取服务器最新 updatedAt
2. 比较：server.updatedAt > record.updatedAt
3. 如果冲突：
   - 方案A（保守）：拒绝同步，提示用户从服务器拉取最新数据
   - 方案B（激进）：服务器数据覆盖本地
   - 建议：默认方案A，允许多设备同步时强制覆盖选项
```

## 九、风险与注意事项

### 9.1 风险

| 风险               | 影响     | 缓解措施                   |
| ---------------- | ------ | ---------------------- |
| 过渡期内联/独立两套并存     | 数据不一致  | 前端迁移完成后统一，后端保留两套接口     |
| Base64 编码问题      | 附件损坏   | 上传/下载使用统一的 Base64 编解码  |
| 大文件上传超时          | 上传失败   | 前端增加上传进度提示，后端增加超时配置    |
| 并发上传同一 file_type | 覆盖而非追加 | 当前设计为覆盖；Phase 5 可改为多文件 |
| 升版后 BOM 子项引用旧版本  | 数据不一致  | 前端升版时提示用户是否同时升版子项      |
| 字典被删除但前端缓存未更新    | 界面异常   | 字典缓存增加版本号，变更时刷新        |
| 字段映射错误导致数据覆盖     | 数据损坏   | 上线前完整测试所有字段            |
| 队列合并导致某些操作丢失     | 功能异常   | 严格测试合并边界条件             |

### 9.2 注意事项

1. **不需要修改数据库**：attachments 表已存在，无需 DDL
2. **无需修改后端**（附件部分）：后端 API（`attachments.py`）已完整实现
3. **前端修改影响面广**：涉及零件/部件的新建、编辑、查看、删除多个页面
4. **测试重点**：确保"修改实体其他字段不影响已有附件"这一核心场景通过
5. **后端修改需重启容器**：`docker restart bom_backend`
6. **前端热更新**：修改 `index.html` 后刷新浏览器即可

## 十、同步与附件的关系

附件功能属于 **独立于通用同步框架的特殊数据类型**，需要单独处理：

| 方面        | 通用同步（Parts/Components） | 附件同步        |
| --------- | ---------------------- | ----------- |
| 数据规模      | 小（KB级）                 | 大（MB级）      |
| 同步方式      | 队列静默同步                 | 独立 API，即时操作 |
| Base64 处理 | 过滤，不传输                 | 独立 API 传输   |
| 列表加载      | 随列表加载                  | 按需加载（不进列表）  |
| 升版处理      | 复制引用关系                 | 不复制（新建）     |
