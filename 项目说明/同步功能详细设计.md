# BOM Tool 数据同步详细设计方案

> 编写日期：2026-04-21
> 版本：v1.1（完善版）

## 一、当前同步方案分析

### 1.1 整体架构

系统采用**本地优先 + 后台静默同步**架构：

```
用户操作 → 立即写入 localStorage → 后台同步队列 → 服务器
                ↓                              ↓
           UI 立即更新                   异步推送/拉取
```

**核心组件：**
- `Store._syncQueue`：待同步任务队列（FIFO）
- `Store._enqueue()`：本地操作完成后自动入队
- `Store._processQueue()`：串行执行同步任务（非阻塞）
- `Store._getIdMap()`/`_setIdMap()`：本地临时ID ↔ 服务器真实ID 映射
- `Store.resolveId()`：解析服务器真实ID
- `Store.registerId()`：注册 ID 映射关系

### 1.2 当前各模块同步方案

#### 1.2.1 零件（Parts）同步

| 操作 | 同步方式 | 冲突检测 |
|------|----------|----------|
| 新建（add） | API.createPart → 服务器生成 UUID | 检查 code+version 是否冲突 |
| 修改（update） | API.updatePart | 比较 updatedAt，服务器更新则拒绝 |
| 删除（delete） | API.deletePart | 无 |
| 升版 | 无（升版视为新建） | 检查 code+version |

**现状：**
- 同步完成后，从服务器拉取最新数据更新本地（覆盖本地）
- 字段映射：`_fieldMap['parts']` 只包含 `status/remark/revisions/createdAt/updatedAt`
- **缺失同步的字段**：name、spec、material_id、price、stock、attachments

#### 1.2.2 部件（Components）同步

| 操作 | 同步方式 | 说明 |
|------|----------|------|
| 新建（add） | API.createAssembly → 添加子项 → 重新拉取 BOM item IDs | 子项通过 `addAssemblyPart` 逐个上传 |
| 修改（update） | API.updateAssembly → **三步同步子项** | 详见 1.2.2.1 |
| 删除（delete） | API.deleteAssembly（级联删除子项） | 后端 cascade 处理 |
| 升版 | 无（升版视为新建） | 检查 code+version |

**1.2.2.1 部件子项三步同步算法（当前实现）：**

```
1. 新增子项：遍历本地 parts，serverChildIds 不存在 → addAssemblyPart
2. 删除子项：遍历服务器 parts，本地 childId 不存在 → removeAssemblyPart(sp.id) 优先，
   或 removeAssemblyPartByChild(sp.child_id, sp.child_type) 回退
3. 修改数量：遍历服务器 parts，数量不同 → 先删除再添加
```

**关键机制：**
- `localChildIds`：通过 `resolveId()` 解析子项 ID 后构建的 Set
- `bomItemId`：服务器 BOM item 的 UUID，存储在本地 `parts[].bomItemId` 字段
- 子项 ID 映射：本地子项的 `componentId/partId` 可能为临时 ID，通过 `resolveId` 解析

**缺失同步的字段：** name、spec、price、attachments

#### 1.2.3 用户（Users）同步

| 操作 | 同步方式 | 冲突检测 |
|------|----------|----------|
| 新建（add） | API.createUser | 检查 username 是否冲突 |
| 修改（update） | API.updateUser | **无冲突检测** |
| 删除（delete） | API.deleteUser | 无 |

**现状：**
- 同步完成后**不拉取**最新数据到本地
- 字段映射：`_fieldMap['users']` 包含 `realName/department/phone/createdAt/updatedAt`
- **缺失同步的字段**：role、status

#### 1.2.4 系统字典（Dictionary）同步

**当前状态：完全未实现**

- `_fieldMap['dict_materials']` 存在但**未在 `_entityMap` 中注册**
- 无 CRUD 同步逻辑
- 字典数据仅在登录后从服务器拉取一次到内存

## 二、问题汇总

### 2.1 字段同步不完整

| 模块 | 缺失字段 | 影响 |
|------|----------|------|
| Parts | name, spec, material_id, price, stock, attachments | 修改这些字段不会同步到服务器 |
| Components | name, spec, price, attachments | 修改这些字段不会同步到服务器 |
| Users | role, status | 修改这些字段不会同步到服务器 |

### 2.2 升版功能缺失

- 升版应创建新版（版本号+1），老版本保留
- 当前升版需要用户手动新建
- 升版后 BOM 子项引用关系需要迁移

### 2.3 冲突检测不完善

- Users 模块无冲突检测
- Parts/Components 的 updatedAt 检测可能存在时区问题

### 2.4 字典同步缺失

- 字典数据（材料等）未实现 CRUD 同步
- `_fieldMap['dict_materials']` 孤立存在

### 2.5 拉取同步缺失

- Users 修改后未从服务器拉取最新数据
- Parts/Components 的 BOM 反查引用未同步更新

### 3.3 各模块同步详细设计

#### 3.3.1 零件（Parts）同步

##### 新建（add）

```
前置条件：code+version 不冲突
步骤：
  1. toApi(record) 转换所有字段（包含 name/spec/material_id/price/stock/attachments）
  2. 移除 id 字段
  3. API.createPart(rest)
  4. registerId('parts', record.id, created.id)
  5. 拉取服务器最新数据：API.getPart(created.id)
  6. 合并到本地（字段覆盖）
```

##### 修改（update）

```
前置条件：updatedAt 冲突检测
步骤：
  1. toApi(record) 转换所有字段
  2. API.updatePart(resolvedId, apiData)
  3. 拉取服务器最新数据：API.getPart(resolvedId)
  4. 合并到本地（字段覆盖）
```

##### 删除（delete）

```
步骤：
  1. API.deletePart(resolvedId)
  2. 从本地 localStorage 移除
  3. 从所有部件的 parts 数组中移除引用（cascade 更新本地）
```

##### 升版（promote）

```
升版流程（新建 + BOM 迁移）：
前置条件：当前版本状态为 "released"
步骤：
  1. 生成新版本号（如 A→B）
  2. 构造新零件记录（除 id 外所有字段复制，version 更新）
  3. toApi(newRecord)，移除 id
  4. API.createPart(rest)
  5. registerId('parts', newLocalId, created.id)
  6. 拉取最新数据
  7. 【关键】更新所有引用老版本零件的部件：
     - 查询所有部件的 parts 数组
     - 将 childType='part' && partId=oldId 的项替换为 newId
     - 将这些部件标记为 _needsUpdate（触发同步）
```

#### 3.3.2 部件（Components）同步

##### 新建（add）

```
步骤：
  1. toApi(record) 转换所有字段（包含 name/spec/price/attachments）
  2. 从 record 提取 parts 数组
  3. API.createAssembly(rest)
  4. registerId('components', record.id, created.id)
  5. 如果有子项：
     5.1. 遍历 parts，对每个子项调用 API.addAssemblyPart
     5.2. 完成后重新获取：API.getAssemblyParts(created.id)
     5.3. 匹配本地子项与服务器返回，建立 bomItemId 映射
     5.4. Store.saveAll('components', localComps)
  6. 拉取服务器最新数据：API.getAssembly(created.id)
  7. 合并到本地
```

##### 修改（update）

```
前置条件：updatedAt 冲突检测
步骤：
  1. toApi(record) 转换所有字段
  2. API.updateAssembly(resolvedId, rest)
  3. 【子项同步】三步算法（保持现有逻辑，优化 bomItemId 使用）
  4. 拉取服务器最新数据：API.getAssembly(resolvedId)
  5. 合并到本地
```

##### 删除（delete）

```
步骤：
  1. API.deleteAssembly(resolvedId)
  2. 从本地 localStorage 移除
  3. 从其他部件的 parts 数组中移除引用（cascade 更新本地）
```

##### 升版（promote）

```
升版流程（新建 + BOM 迁移）：
前置条件：当前版本状态为 "released"
步骤：
  1. 生成新版本号（如 A→B）
  2. 构造新部件记录（所有字段复制，version 更新，清空 status 为 draft）
  3. toApi(newRecord)，移除 id，提取 parts
  4. API.createAssembly(rest)
  5. registerId('components', newLocalId, created.id)
  6. 如果有子项：
     6.1. 遍历 parts，将每个子项添加到新部件
     6.2. 对于子项中引用的零件/部件：
       - 如果子零件本身也需要升版 → 用新版本 ID
       - 否则 → 用原 ID
     6.3. 完成后拉取 BOM item IDs 建立映射
  7. 拉取服务器最新数据
  8. 合并到本地
```

#### 3.3.3 用户（Users）同步

##### 新建（add）

```
前置条件：username 不冲突

  1. toApi(record) 转换所有字段（包含 role/status）
  2. 移除 id 字段
  3. API.createUser(apiData)
  4. registerId('users', record.id, created.id)

  2. API.updateUser(resolvedId, apiData)
  3. 拉取服务器最新数据：API.getUser(resolvedId)
  4. 合并到本地
```

##### 删除（delete）

```
步骤：

  2. 从本地 localStorage 移除
```

#### 3.3.4 系统字典（Dictionary）同步

##### 数据模型

字典数据以 `dict_type`（类型，如 `material`）+ `value`（值）为唯一键。

##### 新建（add）

```
步骤：
  1. toApi(record) 转换字段（dictType/value）
  2. API.createDictionary(apiData)
  3. registerId('dictionary', record.id, created.id)
  4. 拉取服务器最新数据
  5. 合并到内存中的字典缓存
```

##### 修改（update）

```
步骤：
  1. toApi(record) 转换字段
  2. API.updateDictionary(resolvedId, apiData)
  3. 拉取服务器最新数据
  4. 合并到本地缓存
```

##### 删除（delete）

```
步骤：
  1. API.deleteDictionary(resolvedId)
  2. 从本地缓存移除
  3. 需注意：如果被零件引用，应禁止删除或提示
```

##### 同步策略

字典数据量小、修改频率低，建议：
- 登录时全量拉取到内存
- 修改后局部更新内存缓存
- 定期（如每天）全量刷新

### 3.4 冲突检测完善

#### 3.4.1 时间戳冲突

所有 update 操作增加 `updatedAt` 冲突检测：

```
1. 同步前获取服务器最新 updatedAt
2. 比较：server.updatedAt > record.updatedAt
3. 如果冲突：
   - 方案A（保守）：拒绝同步，提示用户从服务器拉取最新数据
   - 方案B（激进）：服务器数据覆盖本地
   - 建议：默认方案A，允许多设备同步时强制覆盖选项
```

#### 3.4.2 唯一键冲突

| 模块 | 唯一键 | 冲突检测 |
|------|--------|----------|
| Parts | code + version | 检查 code+version 组合 |
| Components | code + version | 检查 code+version 组合 |
| Users | username | 检查 username |
| Dictionary | dict_type + value | 检查组合唯一性 |

### 3.5 级联同步处理

当零件或部件被删除/升版时，引用它的部件需要同步更新：

```
cascadeUpdateReferences(entity, oldId, newId):
  1. 获取所有 components
  2. 遍历每个 component 的 parts 数组
  3. 找到 childType 匹配且 childId === oldId 的项
  4. 更新 childId → newId
  5. 将该 component 标记为 _needsUpdate
  6. Store.saveAll('components', updatedComps)
```

### 3.6 同步队列优化

#### 3.6.1 合并同实体多次操作

如果队列中有多次对同一实体的操作，合并为最后一次：

```
合并规则：
- 同实体 + 同 ID 的 add：保留最后一次
- 同实体 + 同 ID 的 update：合并字段（保留所有字段的最新值）
- add 后接 update：合并
- update 后接 delete：只保留 delete
- delete 后接其他：忽略（实体已不存在）
```

#### 3.6.2 优先级

```
高优先级：delete（确保删除优先）
中优先级：add / update
低优先级：级联更新（cascade）
```

## 五、实现优先级

### Phase 1：字段映射完善（高优先级）
1. 补充 `_fieldMap` 缺失字段
2. 确保所有 CRUD 同步包含完整字段

### Phase 2：Users 同步完善（中优先级）
1. 补充 role/status 字段映射
2. 增加 updatedAt 冲突检测
3. 修改后拉取最新数据

### Phase 3：升版功能（高优先级）
1. 实现零件升版（含 BOM 引用迁移）
2. 实现部件升版（含 BOM 引用迁移）
3. 升版时 cascade 更新引用

### Phase 4：字典同步（中优先级）
1. 补充 `_entityMap['dictionary']`
2. 实现 createDictionary / updateDictionary / deleteDictionary
3. 前端字典编辑界面同步

### Phase 5：队列优化（低优先级）
1. 实现操作合并
2. 实现优先级
3. 完善错误提示和重试机制
