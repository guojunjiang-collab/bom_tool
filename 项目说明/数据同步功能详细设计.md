## 一、核心变更说明

### 1.1 字段精简

**删除以下字段**（后续通过自定义字段实现）：

- material_id（材质）
- price（单价）
- stock（库存）
- remark（备注）

**保留字段**：

- 零件：code, name, spec, version, status, revisions
- 部件：code, name, spec, version, status, revisions

### 1.2 附件同步方案

**采用方案B（独立附件表）**：

- 前后端统一使用 attachments 表
- 废弃 parts/assemblies 表的内联附件字段
- 附件通过独立 API 管理

## 二、当前同步架构分析

### 2.1 整体架构

系统采用**本地优先 + 后台静默同步**架构：

```
用户操作 → 立即写入 localStorage → 后台同步队列 → 服务器
                ↓                              ↓
            UI 立即更新                   异步推送
```

**核心组件：**

- `Store._syncQueue`：待同步任务队列（FIFO）
- `Store._enqueue()`：本地操作完成后自动入队
- `Store._processQueue()`：串行执行同步任务
- `Store.resolveId()`：解析服务器真实ID
- `Store.registerId()`：注册本地ID→服务器ID映射

### 2.2 当前支持的实体

| 实体         | create         | update         | delete         | 同步状况 |
| ---------- | -------------- | -------------- | -------------- | ---- |
| parts      | createPart     | updatePart     | deletePart     | ✅    |
| components | createAssembly | updateAssembly | deleteAssembly | ✅    |
| users      | createUser     | updateUser     | deleteUser     | ✅    |

## 六、附件同步流程

### 6.1 上传附件

```
1. 用户选择文件 → Base64 编码
2. 调用 API.uploadAttachment(entity_type, entity_id, file_type, file_name, file_data)
3. 服务器保存到 attachments 表
4. 返回附件 ID
```

### 6.2 下载附件

```
1. 调用 API.listAttachments(entity_type, entity_id) 获取附件列表
2. 显示附件文件名
3. 用户点击下载 → API.getAttachment(attachment_id)
4. 返回 Base64 数据 → 触发下载
```

### 6.3 删除附件

```
1. 调用 API.deleteAttachment(entity_type, entity_id, file_type)
2. 服务器删除 attachments 表记录
```

### 6.4 前端附件 API 封装

```javascript
// api.js
async listAttachments(entity_type, entity_id) {
  return this._fetch('GET', '/attachments/?entity_type=' + entity_type + '&entity_id=' + entity_id);
},

async uploadAttachment(entity_type, entity_id, file_type, file_name, file_data) {
  return this._fetch('POST', '/attachments/', {
    entity_type, entity_id, file_type, file_name, file_data
  });
},

async getAttachment(attachment_id) {
  return this._fetch('GET', '/attachments/' + attachment_id);
},

async deleteAttachment(entity_type, entity_id, file_type) {
  return this._fetch('DELETE', '/attachments/by-entity?entity_type=' + entity_type + '&entity_id=' + entity_id + '&file_type=' + file_type);
},
```

## 十二、Bug 修复记录

### 2026-04-22: 自定义字段值不同步 (多次修复)

**问题1**: 清除缓存后重新登录，自定义字段内容丢失

**原因1**: `onLogin()` 只调用 `syncAll()` 推送本地数据，没有从服务器拉取自定义字段值

**修复1**:

- 在 `store.js` 的 `onLogin()` 中增加 `_loadDataFromServer()` 和 `_loadCustomFieldData()` 调用

---

**问题2**: "检出数据"后自定义字段内容丢失

**原因2**: sync-manager.js 的"检出数据"功能只拉取实体数据，没有加载自定义字段值

**修复2**:

- 在 sync-manager.js 的检出数据流程中添加加载自定义字段值的步骤

---

**问题3**: 新建零件/部件时自定义字段未上传到服务器

**原因3**: 保存自定义字段时使用的是本地ID，但服务器创建后会返回新的服务器ID，需要等待ID映射完成

**修复3**:

- 修改 `_saveCFValues()` 函数，增加重试机制，等待 `Store.resolveId()` 返回服务器ID后再保存

---

**问题4**: 清除缓存时数据正在上传

**原因4**: 用户在数据上传过程中点击"清除缓存"，导致数据丢失

**修复4**:

- 在 sync-manager.js 的清除缓存按钮点击事件中，检查 `Store._syncRunning` 和 `Store._syncQueue`，如有正在进行的同步则弹窗提醒

## 十三、附件功能优化 v3.0

### 13.1 优化目标

- **前端加载速度**：不将附件实体下载到本地，只显示附件名称
- **按需下载**：用户点击下载时才通过 attachment_id 获取附件内容
- **节省存储**：使用 PostgreSQL BYTEA 替代 Base64，节省 33% 存储空间
