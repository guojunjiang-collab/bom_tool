## 一、需求目标

### 业务背景
在BOM管理系统中，零件和部件的基础字段（件号、名称、规格、材料等）是固定的，无法满足不同企业/项目的特殊信息记录需求。需要提供自定义字段功能，让管理员可按需扩展零部件属性。

### 功能范围
1. **字段定义**：管理员可定义自定义字段的名称、类型（文本/数字/单选/多选）、是否必填、选项列表
2. **数据存储**：字段定义和字段值分别存储，支持查询与筛选
3. **界面展示**：
   - 零件/部件详情页：只读展示自定义字段值
   - 零件/部件编辑页：可编辑的表单控件
   - 系统设置页：自定义字段定义的管理界面（增删改排序）

## 三、详细设计方案

### 3.1 数据库设计

#### 表1：custom_field_definitions（自定义字段定义）

```sql
CREATE TABLE custom_field_definitions (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name VARCHAR(128) NOT NULL,           -- 字段显示名称，如"供应商"
    field_key VARCHAR(64) NOT NULL UNIQUE, -- 字段标识键，如"supplier"，用于数据存储
    field_type VARCHAR(32) NOT NULL,       -- 字段类型：text / number / select / multiselect
    options JSONB DEFAULT '[]',            -- 单选/多选的选项列表，如 ["供应商A","供应商B"]
    is_required BOOLEAN DEFAULT FALSE,     -- 是否必填
    applies_to VARCHAR(32) NOT NULL DEFAULT 'both', -- 适用对象：part / component / both
    sort_order INTEGER DEFAULT 0,          -- 排序序号
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

#### 表2：custom_field_values（自定义字段值）

```sql
CREATE TABLE custom_field_values (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    field_id UUID NOT NULL REFERENCES custom_field_definitions(id) ON DELETE CASCADE,
    entity_type VARCHAR(32) NOT NULL,  -- 'part' 或 'component'
    entity_id UUID NOT NULL,           -- 零件或部件的ID
    value_text TEXT,                    -- 文本/单选值
    value_number DECIMAL(12,4),         -- 数字值
    value_json JSONB,                   -- 多选值数组
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(field_id, entity_type, entity_id)
);

CREATE INDEX idx_cfv_entity ON custom_field_values(entity_type, entity_id);
CREATE INDEX idx_cfv_field ON custom_field_values(field_id);
```

**设计说明**：
- 字段定义与值分离，EAV模式灵活扩展
- `field_key` 作为唯一标识，避免用自增ID暴露给前端
- 值表按类型分列存储（text/number/json），便于查询和类型校验
- `applies_to` 控制字段对零件/部件/两者生效

### 3.2 后端API设计

#### 字段定义 CRUD

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/custom-fields/ | 获取所有字段定义 | admin/engineer |
| GET | /api/custom-fields/?applies_to=part | 按适用对象筛选 | admin/engineer |
| POST | /api/custom-fields/ | 创建字段定义 | admin |
| PUT | /api/custom-fields/{id} | 更新字段定义 | admin |
| DELETE | /api/custom-fields/{id} | 删除字段定义（级联删除值） | admin |
| PUT | /api/custom-fields/reorder | 批量更新排序 | admin |

#### 字段值 CRUD

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/custom-field-values/{entity_type}/{entity_id} | 获取实体的所有自定义字段值 | admin/engineer |
| PUT | /api/custom-field-values/{entity_type}/{entity_id} | 批量设置实体的自定义字段值 | admin/engineer |

**批量设置请求体**：
```json
{
  "values": [
    {"field_id": "uuid-xxx", "value": "供应商A"},
    {"field_id": "uuid-yyy", "value": 100},
    {"field_id": "uuid-zzz", "value": ["选项1", "选项2"]}
  ]
}
```

### 3.3 前端设计

#### 3.3.1 系统设置页 - 自定义字段管理

在现有"字典管理"卡片下方新增"自定义字段管理"卡片：
- 表格展示所有字段定义（名称、标识、类型、适用对象、必填、排序）
- 支持：新增、编辑、删除、上下移动排序
- 编辑弹窗：字段名称、字段标识（创建后不可改）、类型选择、选项管理（单选/多选时）、适用对象、是否必填

#### 3.3.2 零件/部件详情页 - 自定义字段展示

在现有基础信息下方新增"自定义属性"区域：
- 仅显示该实体有值的自定义字段
- 只读展示，文本直接显示、数字格式化、单选显示标签、多选显示标签组

#### 3.3.3 零件/部件编辑页 - 自定义字段编辑

在现有表单下方、备注之前新增自定义字段区域：
- 根据字段定义动态渲染表单控件：
  - text → `<input type="text">`
  - number → `<input type="number">`
  - select → `<select>` 下拉
  - multiselect → 复选框组
- 必填字段加红星标记
- 保存时收集自定义字段值，一并提交

## 五、风险与应对

| 风险项 | 可能性 | 影响度 | 应对措施 |
|--------|--------|--------|----------|
| 字段定义删除后已有数据丢失 | 中 | 中 | 删除前提示确认，级联清理 |
| 大量自定义字段影响性能 | 低 | 低 | 限制字段数量上限（20个），值表建索引 |
| 自定义字段值同步复杂度 | 中 | 中 | 与现有同步队列机制复用 |
| 前端表单动态渲染兼容性 | 低 | 低 | 统一渲染函数，复用已有表单模式 |

## 六、与旧方案的区别

**旧方案（v1.0 2026-04-20）**：尝试直接在压缩的 pages.js 中注入代码，因字符串拼接复杂度过高而失败。

**新方案（v2.0 2026-04-22）**：
- 前端已拆分为独立 JS 文件（pages-parts.js / pages-components.js / pages-settings.js）
- 后端模块化结构清晰，新增路由/模型/Schema 不侵入已有代码
- 采用 EAV 数据模型，值按类型分列存储，避免全 JSONB 查询困难
- 自定义字段数据随零部件数据一起加载，减少额外请求
